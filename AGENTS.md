# AGENTS.md

This project is a website built with **Edge Delivery Services** in Adobe Experience Manager Sites as a Cloud Service, using **AEM Authoring with the Universal Editor** (xwalk). As an agent, follow the instructions in this file to deliver code based on Adobe's standards for fast, easy-to-author, and maintainable web experiences.

The source site being migrated is **https://www.allianz.com.au/**.

---

## Project Overview

This project is based on the [aem-boilerplate-xwalk](https://github.com/adobe-rnd/aem-boilerplate-xwalk) project template, set up following the [Universal Editor tutorial](https://www.aem.live/developer/ue-tutorial). It uses the **Universal Editor** for WYSIWYG content authoring in AEM as a Cloud Service, with Edge Delivery Services for rendering and delivery.

This is **NOT** a Document Authoring (DA) project. Content is authored in AEM via the Universal Editor (not Google Docs/SharePoint). The content source is the AEM Author instance, and content is stored in JCR (Java Content Repository), not as `.plain.html` or `.md` files.

### Key Technologies

- Edge Delivery Services for AEM Sites (documentation at https://www.aem.live/)
- **Universal Editor** for WYSIWYG authoring (documentation at https://www.aem.live/developer/ue-tutorial and https://www.aem.live/developer/universal-editor-blocks)
- **Component modeling** via JSON definition files (`component-definition.json`, `component-models.json`, `component-filters.json`)
- Vanilla JavaScript (ES6+), no transpiling, no build steps
- CSS3 with modern features, no Tailwind or other CSS frameworks
- HTML5 semantic markup generated by the AEM delivery backend, decorated by our code
- Node.js tooling (including `merge-json-cli` for compiling component models)

## Setup Commands

- Install dependencies: `npm install`
- Start local development: `npx -y @adobe/aem-cli up --no-open --forward-browser-logs` (run in background, if possible)
  - Install the AEM CLI globally by running `npm install -g @adobe/aem-cli` then `aem up` is equivalent to the command above
  - The dev server runs at `http://localhost:3000` with auto-reload
- **Build component models**: `npm run build:json` — Compiles distributed `_*.json` model files from `models/` and `blocks/*/` into the root-level `component-definition.json`, `component-models.json`, and `component-filters.json`
- Run linting before committing: `npm run lint`
- Auto-fix linting issues: `npm run lint:fix`

**IMPORTANT**: After modifying any `_*.json` model file (in `models/` or `blocks/*/`), always run `npm run build:json` to recompile the root-level JSON files. The root-level files are what AEM and the Universal Editor actually consume.

## Code Style Guidelines

### JavaScript

- Use ES6+ features (arrow functions, destructuring, etc.)
- Follow Airbnb ESLint rules (configured) plus `eslint-plugin-xwalk` rules
- Always include `.js` file extensions in imports
- Use Unix line endings (LF)
- **Always use `moveInstrumentation()`** when replacing or restructuring DOM elements in block decoration (see Block Development section)

### CSS

- Follow Stylelint standard configuration
- Use modern CSS features (CSS Grid, Flexbox, CSS Custom Properties)
- Maintain responsive design principles
  - Declare styles mobile first, use `min-width` media queries at 600px/900px/1200px for tablet and desktop
- Ensure all selectors are scoped to the block
  - Bad: `.item-list`
  - Good: `.{blockname} .item-list`
- Avoid classes `{blockname}-container` and `{blockname}-wrapper` as those are used on sections
- Never use `!important` — increase selector specificity instead
- Use CSS custom properties — reference design tokens from `PROJECT.md`
- Edge-to-edge blocks — use `:has()` on wrapper: `main > div:has(.block-name)`
- Visually hidden text — use `clip-path: inset(50%)` instead of deprecated `clip`
- Backdrop filter — include both `-webkit-backdrop-filter` and `backdrop-filter`
- Avoid fragile selectors — don't depend on sibling element sequences. Prefer block/section variants with explicit class names
- Use `box-sizing: border-box` when setting explicit width/height on elements with padding

### HTML

- Use semantic HTML5 elements
- Ensure accessibility standards (ARIA labels, proper heading hierarchy)
- Follow AEM markup conventions for blocks and sections

## Key Concepts

### Content

Content is authored in **AEM as a Cloud Service** using the **Universal Editor**, a WYSIWYG editor that provides in-context editing. Content is stored in **JCR** (Java Content Repository) and delivered to Edge Delivery Services via the Franklin delivery servlet.

The content model is defined by the component definition, model, and filter JSON files. Authors create pages by adding sections and blocks from a component picker. Each block's editable properties are defined in `component-models.json` and exposed in the Universal Editor's properties panel.

Background on content and markup structure can be found at https://www.aem.live/developer/markup-sections-blocks and https://www.aem.live/developer/markup-reference.

You can inspect the contents of any page with `curl http://localhost:3000/path/to/page` and `curl http://localhost:3000/path/to/page.plain.html`.

### Component Modeling

Every block, section, and default content component must be defined in three JSON files. These are the compiled root-level files consumed by the Universal Editor:

1. **`component-definition.json`** — Declares available components, their AEM resource types, and default templates
2. **`component-models.json`** — Defines the editable fields (properties panel) for each component
3. **`component-filters.json`** — Controls which components can be placed inside which containers

The source files are distributed across `models/` (shared components) and `blocks/*/_*.json` (block-specific). They are compiled by `npm run build:json` using `merge-json-cli`.

#### Component Definition Pattern

```json
{
  "title": "My Block",
  "id": "my-block",
  "plugins": {
    "xwalk": {
      "page": {
        "resourceType": "core/franklin/components/block/v1/block",
        "template": {
          "name": "My Block",
          "model": "my-block"
        }
      }
    }
  }
}
```

- `resourceType`: Always `core/franklin/components/block/v1/block` for blocks, `core/franklin/components/block/v1/block/item` for child items
- `template.name`: The block name as it appears in EDS markup
- `template.model`: References a model ID from `component-models.json`
- `template.filter`: References a filter ID from `component-filters.json` (for container blocks)

#### Container Blocks (with child items)

Container blocks (like Cards, Carousel, Tabs) define two components:
- The **container** (`resourceType: .../block`, with `filter` pointing to a filter that allows the item component)
- The **item** (`resourceType: .../block/item`, with `model` for per-item fields)

#### Field Types in Component Models

| `component` value | Purpose | Key properties |
|-------------------|---------|----------------|
| `text` | Single-line text input | `valueType` (string/number), `multi` (boolean for tag-like input) |
| `richtext` | Rich text editor (WYSIWYG) | `valueType: "string"` |
| `reference` | Asset picker (DAM images) | `multi: false` for single asset |
| `select` | Dropdown single-select | `options[]` with `name`/`value` pairs |
| `multiselect` | Multi-select checkboxes | `options[]` with `name`/`value` pairs |
| `aem-content` | AEM content/page reference picker | For links and fragment references |
| `aem-tag` | AEM tag picker | For taxonomy tags |
| `boolean` | Toggle/checkbox | Returns `true`/`false` |
| `number` | Numeric input | For quantities, dimensions |
| `date-time` | Date/time picker | ISO8601 format |
| `tab` | Visual tab separator | Groups fields into tabs in the properties panel |
| `container` | Composite multi-field | `multi: true` with nested `fields[]` |

#### Field Collapse (Property Name Conventions)

Multiple properties collapse into semantic HTML elements based on naming suffixes:

- **Images**: `image` + `imageAlt` → `<picture><img src="..." alt="...">`
- **Links/Buttons**: `link` + `linkText` + `linkTitle` + `linkType` → `<a href="..." title="...">text</a>` with type class
- **Headings**: `title` + `titleType` → `<h2>Title</h2>` (type sets heading level)

#### Element Grouping (Underscore Convention)

Properties prefixed with a group name and underscore (e.g., `teaserText_title`, `teaserText_description`) are grouped into a single cell in the rendered table markup. Field collapse rules apply within groups.

#### Block Options via `classes` Property

A `classes` field (using `select` or `multiselect` component) lets authors choose block variants/styling:

```json
{
  "component": "select",
  "name": "classes",
  "label": "Style",
  "valueType": "string",
  "options": [
    { "name": "Default", "value": "" },
    { "name": "Highlight", "value": "highlight" }
  ]
}
```

This maps to CSS classes on the block element: `<div class="my-block highlight">`.

### Blocks

Blocks add styling and functionality to content. Each block has:
- A **component model** (`_*.json`) defining its editable fields and structure
- A **JavaScript file** (`blockname.js`) that decorates the delivered HTML
- A **CSS file** (`blockname.css`) for styling

The block JavaScript exports a default function called to perform decoration:

```javascript
import { moveInstrumentation } from '../../scripts/scripts.js';

export default async function decorate(block) {
  // 1. Load dependencies
  // 2. Extract configuration from block.children
  // 3. Transform DOM (ALWAYS use moveInstrumentation when replacing elements)
  // 4. Add event listeners
}
```

**CRITICAL**: When restructuring DOM in block JS (e.g., converting `<div>` rows to `<li>` elements), you **MUST** call `moveInstrumentation(oldElement, newElement)` to transfer `data-aue-*` and `data-richtext-*` attributes. Without this, the Universal Editor loses the ability to edit those elements in-context.

Use `curl` and `console.log` to inspect the HTML delivered by the backend and the DOM nodes to be decorated before making assumptions. Remember that authors may omit or add fields to a block, so your code must handle this gracefully.

Each block should be self-contained and reusable, with CSS and JS files following the naming convention: `blockname.css`, `blockname.js`, `_blockname.json`.

### moveInstrumentation

The `moveInstrumentation(from, to)` function (exported from `scripts.js`) is the single most important UE-specific pattern. It transfers Universal Editor instrumentation attributes (`data-aue-*` and `data-richtext-*`) from one DOM element to another.

**When to use it:**
- When replacing a row `<div>` with a `<li>`, `<tr>`, `<details>`, or any other semantic element
- When replacing an `<img>` with an optimized `<picture>` element
- When moving content from a block child to a new wrapper element
- Essentially: any time you discard an original DOM element and create a new one in its place

**Example** (from the Cards block):
```javascript
import { createOptimizedPicture } from '../../scripts/aem.js';
import { moveInstrumentation } from '../../scripts/scripts.js';

export default function decorate(block) {
  const ul = document.createElement('ul');
  [...block.children].forEach((row) => {
    const li = document.createElement('li');
    moveInstrumentation(row, li);  // Transfer UE attributes
    while (row.firstElementChild) li.append(row.firstElementChild);
    // ... classify children ...
    ul.append(li);
  });
  ul.querySelectorAll('picture > img').forEach((img) => {
    const optimizedPic = createOptimizedPicture(img.src, img.alt, false, [{ width: '750' }]);
    moveInstrumentation(img, optimizedPic.querySelector('img'));  // Transfer on images too
    img.closest('picture').replaceWith(optimizedPic);
  });
  block.replaceChildren(ul);
}
```

### Auto-Blocking

Auto-blocking is mostly **disabled** in xwalk projects because content is explicitly modeled through component definitions. The `buildAutoBlocks` function in `scripts.js` is an empty placeholder. Authors add blocks explicitly via the Universal Editor's component picker.

Only add auto-blocking for very specific cases (e.g., modals from link patterns) where it genuinely improves the authoring experience.

### Three-Phase Page Loading

Pages are progressively loaded in three phases to maximize performance. This process begins when `loadPage` from `scripts.js` is called:

- **Eager** — Load only what is required to get to LCP. This includes decorating sections, blocks, buttons, icons, and loading the first section.
- **Lazy** — Load all other page content, including the header and footer.
- **Delayed** — Load things that can be safely loaded later (martech, analytics, etc.).

## Critical Rules

1. **Always read files before editing** — Never modify code without reading it first.
2. **REUSE existing blocks** — Always check the Block Reference in `PROJECT.md` before creating new blocks or variants.
3. **Keep `PROJECT.md` up-to-date** — Update it when creating/modifying/deleting blocks, variants, section styles, tokens, component models, or import infrastructure.
4. **Create variants, not new blocks** — When a content pattern is similar to an existing block but needs different styling, create a VARIANT of that block (not a new block). Variants are implemented via the `classes` property in the component model.
5. **Always use `moveInstrumentation()`** — When restructuring DOM elements in block JS, transfer UE instrumentation attributes. Without this, in-context editing breaks.
6. **Always run `npm run build:json` after modifying `_*.json` files** — The root-level compiled JSON files are what the Universal Editor consumes.
7. **Never import all-caps content as-is** — Convert to Title Case or Sentence case in content; apply `text-transform: uppercase` via CSS.
8. **Don't rely on bold/strong for block-wide styling** — Apply `font-weight: 700` via CSS. Reserve `<strong>` only for inline emphasis.
9. **NEVER commit or push to Git yourself** — The user handles all Git operations.
10. **Test in preview** — Verify changes at `http://localhost:3000`.
11. **Merge similar blocks** — Prefer single multi-row blocks over separate blocks per row.
12. **Never edit compiled JSON files directly** — Always edit the source `_*.json` files in `models/` or `blocks/*/` and recompile with `npm run build:json`.
13. **Content lives in AEM (JCR), code lives in Git** — Never push content HTML via Git. Never modify `.gitignore` to track HTML content.
14. **Check `PAGES.txt` before modifying ANY parser** — Review `/PAGES.txt` to understand which pages may be affected. Flag concerns to the user before proceeding.
15. **Fragment files (nav, footer)** — Must NOT have `<header>` or `<footer>` tags in their content.

## Content Architecture

### Strict Separation: Content in AEM (JCR), Code in Git

- **Code** (JS, CSS, JSON models, config): Lives in Git, deployed via AEM Code Sync
- **Content** (pages, fragments, assets): Lives in AEM as a Cloud Service (JCR), authored via the Universal Editor, previewed/published via AEM admin API
- **fstab.yaml**: Points to the AEM Author instance with `type: "markup"` and `suffix: ".html"`

**Rules:** Never push content HTML via Git. Code (including component model JSON files) is committed to Git. Content is managed in AEM.

### UE Markup Compatibility

The Universal Editor decorates page content with **instrumentation attributes** (`data-aue-*` and `data-richtext-*`). These attributes enable in-context editing. When block JavaScript restructures the DOM, these attributes must be transferred to the new elements via `moveInstrumentation()`.

The xwalk version of `aem.js` includes guards against double-decoration (checking `data-section-status` and `data-block-status`), and `wrapTextNodes()` automatically preserves instrumentation when wrapping text in `<p>` tags.

### Editor Support (Live Editing)

The `scripts/editor-support.js` file enables live editing in the Universal Editor. When an author changes a property in the UE properties panel:
1. The UE emits events (`aue:content-patch`, `aue:content-update`, etc.)
2. `editor-support.js` receives the updated HTML, sanitizes it with DOMPurify, and patches the DOM in-place
3. Blocks are re-decorated as needed

You should **never modify `editor-support.js`** unless fixing a specific live-editing bug.

## Block Development

### Before Creating a New Block

1. **Check the Block Reference in `PROJECT.md`** — Review all existing blocks and their variants
2. **Analyze if existing blocks can work** — Consider variants (via `classes` property), section styles, or new `classes` options
3. **Only create new blocks when** — No existing block can accommodate the content, structure is fundamentally different, or a variant would require >50% new code

### Decision Tree for Content Mapping

```
New content section identified
    |
Does it match an existing block's purpose?
    |-- YES -> Use that block (or add a variant via `classes` option)
    |-- NO  -> Is it similar to any existing block?
                |-- YES -> Create new VARIANT (add `classes` option + CSS)
                |-- NO  -> Create new BLOCK (full _*.json + JS + CSS, document in PROJECT.md)
```

### Creating a New Block (Checklist)

1. Create the block directory: `blocks/{blockname}/`
2. Create the model file: `blocks/{blockname}/_{blockname}.json` with:
   - `definitions[]` — Component definition(s) for UE component picker
   - `models[]` — Field definitions for UE properties panel
   - `filters[]` — Containment rules (if container block)
3. Create `blocks/{blockname}/{blockname}.js` — Block decoration logic (use `moveInstrumentation`)
4. Create `blocks/{blockname}/{blockname}.css` — Block styles
5. Update `models/_section.json` → Add the block ID to the section filter's `components` array
6. Run `npm run build:json` to recompile root-level JSON files
7. Document the block in `PROJECT.md`

### Variant Implementation (via `classes`)

Instead of creating separate blocks for visual variations, add a `classes` field to the block's model:

```json
{
  "component": "select",
  "name": "classes",
  "label": "Style",
  "valueType": "string",
  "options": [
    { "name": "Default", "value": "" },
    { "name": "Featured", "value": "featured" },
    { "name": "Compact", "value": "compact" }
  ]
}
```

Then add variant CSS in the same block's CSS file:

```css
.cards.featured .cards-card-body { /* featured variant styles */ }
.cards.compact .cards-card-body { /* compact variant styles */ }
```

Update block JS only if the variant requires different DOM structure.

### Variant Naming Convention

- **Block-specific variants**: Use descriptive names (e.g., `featured`, `compact`, `with-icon`)
- **Section styles**: Applied via the section model's `style` multiselect field

## Migration Rules

### Skill Workflow / Migration Order

Respect Experience Modernization Agent skill dependencies:

1. **Bulk import requires import infrastructure** — Migrate at least one representative page per template before running bulk import. Single-page migration creates page templates, parsers, and transformers that bulk import reuses.
2. **Block styling (phase 2) requires site-wide design (phase 1)** — Block CSS references global design tokens in `styles.css`. When migrating design, complete site-wide tokens before styling individual blocks.
3. **When migrating** — Map content to block variants, create import infrastructure (templates, parsers, transformers), define component models, and document everything in `PROJECT.md`.

### Wide Viewport for Content Extraction

**Always set the browser viewport to wide desktop (>=1400px width) before extracting content from source pages.** Responsive images, background images, and some content (mega menus, "Show More") are only correct at desktop widths.

### Import Infrastructure

Import infrastructure for xwalk projects includes:
- **Page templates** (`page-templates.json`) — Define section/block structure for each template type
- **Parsers** (`tools/importer/parsers/`) — Extract content from source HTML into block structures
- **Transformers** (`tools/importer/transformers/`) — Clean up and normalize source HTML before parsing
- **Import scripts** — Bundle parsers and transformers for execution

Content files generated by the import pipeline are `.plain.html` files used for local preview during development. The final content will be authored in AEM via the Universal Editor using the component models defined in this project.

### Block Reference in Import Context

When designing import parsers, ensure the output HTML structure matches what the component model expects. Each block's `_*.json` model defines the fields and their types — the parser should produce table markup that maps to those fields.

## Image URL Rules

### AEM Assets

In xwalk projects, images are stored in AEM Assets (DAM). Image references in component models use the `reference` field type, which allows authors to pick images from the DAM via the Universal Editor.

For migration, images from the source site need to be uploaded to AEM Assets. The import process should reference the original image URLs; final image management happens in AEM.

### Local Preview

When the dev server runs, local `.plain.html` files are served at `http://localhost:3000/content/<page-path>`. Images in these files can reference external URLs for preview purposes.

## Deployment

### Environments

Your local development server at `http://localhost:3000` serves code from your local working copy (even uncommitted code) and content that has been previewed by authors.

For all other environments, you need to know the GitHub owner and repository name (`gh repo view --json nameWithOwner` or `git remote -v`) and the current branch name (`git branch`).

- **Production Preview**: `https://main--{repo}--{owner}.aem.page/`
- **Production Live**: `https://main--{repo}--{owner}.aem.live/`
- **Feature Preview**: `https://{branch}--{repo}--{owner}.aem.page/`

### Publishing Process

1. Push changes to a feature branch
2. AEM Code Sync automatically processes changes making them available on the feature preview environment for that branch
3. Run a PageSpeed Insights check against the feature preview URL. Target a score of 100
4. Open a pull request to merge changes to `main`
   1. In the PR description, include a link to `https://{branch}--{repo}--{owner}.aem.page/{path}` with a path to a page that illustrates the change
   2. If no existing page exists, create test content as a static HTML file and ask the user for help copying it to a CMS content page you can link in the PR
5. Use `gh pr checks` to verify the status of code synchronization, linting, and performance tests
6. A human reviewer will review the code, inspect the provided URL, and merge the PR
7. AEM Code Sync updates the main branch for production

## Security Considerations

- Never commit sensitive information (API keys, passwords)
- Consider that everything you do is client-side code served on the public web
- Follow Adobe security guidelines
- Regularly update dependencies
- Use the `.hlxignore` file to prevent files from being served (same format as `.gitignore`)
- The `head.html` includes a Content Security Policy with nonce-based script loading (`nonce-aem`) required for Universal Editor integration
